////////////////////////////////////////////////////////////////////////////

- In api-gateway media module
-- Controller
@Get('test')
test( @CurrentUser() user: UserDto, @Req() req: Request) {
const cookie = req.cookies?.Authentication                 ;
return this.mediaService.test(cookie)                      ;
}

-- Service
async test(authCookie: string) {
return await firstValueFrom(
this.mediaClient.send<any, any>('media_test', {
Authentication: authCookie // ===>> IMPORTANT: NOTE - 01 below
}),
)                                                              ;
}

- In media microservice
-- Controller
@UseGuards(JwtAuthGuard)
@MessagePattern('media_test')
test(@Payload() payload: any, @CurrentUser() user: UserDto): any {
return {message: `REACHED MEDIA TEST`, payload, user}              ;
}

NOTE - 01:
- If we dont send Authentication: authCookie, th media_test consumer will give an error (in the media microservice).
- The error will be this {status: 'error', message: 'No JWT provided'}
- The Authentication: cookie format is because of our common JWtAuthguard code check the handleRpc section.
- This wholow flow is how we can get current user when using consumers and also add protection to consumers.
- However, for cases such as the case of our worker emitting event, the worker does not have a cookie (user passes cookie through browser but of course a worker doesnt have access)
- For that case, we can skip the JwtAuthGuard on the consumer function, and if user id is needed it can be passed through from the worker.
- The aspect of not putting a guard over the consumer function doesn't seem perfect, but had a discussion with claude in which it clarified some doubts. Will look into this further.

UPDATE IMPORTANT:
Kafka consumer functions aparantly shouldnt be authenticated, as per chatgpt
A Kafka consumer isn't receiving an HTTP request. It's receiving a message from another service.

By the time the event is published, the gateway has already:

Validated the cookie
Identified the user
Authorized the action (if necessary)

The consumer's job is to process a trusted event, not authenticate a user.

In order to pass the userId, the most common approach is to pass it from the API gateway via the payload
Once the gateway authenticates the request and extracts the user from the cookie, it should include the user information that downstream services need in the Kafka message.

One To One Explicit Id
////////////////////////////////////////////////////////////////////////////
Reference: https://www.youtube.com/watch?v=rKgZLVgdvAY&t=333s
Explicitly setting the mediaId column comes down to TypeORM's internal management versus direct access in your code.

The Implicit Approach (Standard)
When you use @JoinColumn(), TypeORM automatically creates a hidden mediaId column in your database table to serve as the foreign key. You don't have to define this property in your TypeScript class, and TypeORM handles it entirely behind the scenes when saving or joining entities.

The Explicit Approach (Manual Property)
By manually adding mediaId as a property in your entity, you gain a few practical benefits:

Direct Access: You can read the ID directly from the object (e.g., comment.mediaId) without needing to load the entire media relationship object first.
Efficiency: If you only need the ID, you can perform logic without triggering an expensive JOIN query to fetch the full Media entity from the database.
Clarity/Debugging: Having the column explicitly defined in your entity class makes it easier to see exactly what columns exist in your database table just by looking at your code.
Key Takeaway: You don't need to define it explicitly for the relationship to function—TypeORM will manage the foreign key regardless. Adding it manually is simply a design choice for when you want easier, more performant access to that foreign key value within your application logic.

loadRelationCountAndMap lets you attach a computed count of a relation onto an entity result without changing your database schema.
For exmple: "give me a number based on a relation and stick it on the returned object"

const comments = await this.commentRepository
.createQueryBuilder('comment')
.loadRelationCountAndMap(
'comment.replyCount',
'comment.replies'
)
.getMany()                                    ;
